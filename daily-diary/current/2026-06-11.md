   - The index table was only showing 4 columns: NOR Rate, OT Rate, SUN Rate, PUB HOL. But `sun_ot_rate` and `pub_ot_rate` were missing from the display entirely (DB had all 6 the whole time).
   - Rebuilt the table header with a 2-row structure matching the Excel: 3 groups (Unit Rate / Sun Rate / Pub. Hol) each with NOR and O/T sub-columns — now all 6 columns visible.

3. **PDF section ordering**:
   - `$groupedItems` was just `groupBy('section_label')` — no defined order. PDF sections appeared in whatever order DB rows happened to come back.
   - Fixed: sections now sorted Indirect Manpower → Direct Manpower → Indirect Manpower (OVERTIME) → Direct Manpower (OVERTIME) → everything else.
   - Within each section: rows sorted by `qty × unit_price` descending — most expensive item on top.
   - Applied same logic to `show` view for consistency.

4. **PDF typed row expansion — the main feature of today**:
   - Previously a manpower entry like "Maintenance Planner — 20 NOR + 2 SUN + 2 PH" saved as one blended row (24 days @ 681.44). In the PDF it looked like all 24 days were the same rate — no visibility into the type breakdown.
   - Fixed: PDF now expands each manpower row using the stored `meta` and the `rateItem` relationship, showing separate lines per type:
     - Regular rows → NOR / SUN / PH with actual `nor_rate`, `sun_nor_rate`, `pub_nor_rate`
     - OT rows → NOR O/T / SUN O/T / PH O/T with actual `ot_rate`, `sun_ot_rate`, `pub_ot_rate`
   - Non-manpower rows (Scaffolding, T&C%, Cost Plus) render as single lines unchanged.
   - Old OT rows saved before this change (meta: `{ot_hours}`) gracefully fall through to single-row display — no breakage.

5. **Smart Add — OT inputs split into 3**:

   - `confirmAddManpower()` now computes a blended OT rate from all 3 types and stores meta as `{nor_ot_hours, sun_ot_hours, pub_ot_hours}`.

## Key Insights

- Running the same project on two machines is a common source of "DB out of sync" bugs. The middleware and migration issues today were both caused by this. Worth always checking `php artisan migrate:status` when moving machines.
- The rate card structure from the Excel (6 columns, 3 groups × NOR/OT) was always in the DB — the UI just wasn't surfacing it properly. Simple display fix with big clarity impact.
- Expanding blended rows in the PDF view (instead of storing separately) is a good pattern: single row in DB (clean storage), full breakdown in PDF (client-facing transparency). The rounding difference is a known trade-off.
- Subagents for code-level verification are effective — the smoke test result was detailed and caught the blending artefact independently.
